# AI Dev Rules

이 문서는 팀원이 ChatGPT, Codex, Cursor 같은 AI 도구에 작업을 맡길 때 공통으로 붙여넣는 기준입니다.

## 공통 프로젝트 설명

우리는 `Brainrot Patent`라는 2주 MVP 데모를 만든다. 사용자가 황당한 발명 아이디어를 한국어로 입력하면, AI가 특허 명세서처럼 보이는 7개 섹션 문서로 변환한다. 결과물은 실제 특허 자문이 아니라 엔터테인먼트 목적이다.

## 절대 지켜야 할 범위

- 로그인, 결제, 사용자 DB는 만들지 않는다.
- 실제 법률 자문처럼 표현하지 않는다.
- 신규성/진보성 판단 기능을 만들지 않는다.
- API 스키마는 `docs/API_CONTRACT.md`를 따른다.
- 한국어 MVP만 지원한다.
- 데모 안정성이 기능 수보다 우선이다.

## 코드 스타일

- TypeScript와 Python 타입 힌트를 최대한 사용한다.
- 에러 응답은 프론트엔드가 처리하기 쉽게 JSON으로 반환한다.
- 임의로 필드명을 바꾸지 않는다.
- 외부 API 키는 코드에 하드코딩하지 않는다.
- `.env.example`에는 키 이름만 넣는다.

## AI에게 작업을 맡길 때 기본 프롬프트

```text
너는 2주짜리 대학생 팀 프로젝트의 개발 보조자다.
프로젝트는 Brainrot Patent이며, 황당한 한국어 발명 아이디어를 특허 명세서처럼 보이는 7개 섹션 문서로 변환하는 AI 에이전트 데모다.

중요 제약:
- 실제 특허 출원/법률 자문처럼 만들지 말 것
- 한국어 입력/출력만 지원
- 로그인/결제/DB는 만들지 말 것
- API 스키마는 docs/API_CONTRACT.md를 따를 것
- 데모 안정성이 최우선

내가 맡은 역할은 [역할명]이다.
지금 작업은 [구체 작업]이다.
기존 문서와 충돌하지 않게 최소 변경으로 구현해줘.
```

## 구현 판단 기준

무엇을 만들지 애매하면 다음 순서로 판단합니다.

1. 발표장에서 보이는가?
2. 2주 안에 끝나는가?
3. 다른 팀원의 인터페이스를 깨지 않는가?
4. 실패했을 때 fallback을 줄 수 있는가?
